# 实验一：数据库概念结构设计

## 一、实验目的

1. 能根据实际业务需求抽象出实体、实体的属性和实体间的联系及联系的属性
2. 能根据E-R图的绘制规范，运用E-R图描述局部业务数据关系
3. 能将局部E-R图合并优化形成全局E-R图

---

## 二、系统概述

### 2.1 项目名称
英超球员 2019-2023 赛季表现数据管理系统

### 2.2 需求描述
- **球员基本信息管理**：管理球员的姓名、国籍、位置、出生日期等基本信息
- **赛季数据管理**：管理2019-2023五个赛季的比赛安排信息
- **进攻与技术指标管理**：管理球员的进球、助攻、射门等进攻数据
- **防守指标管理**：管理球员的抢断等防守数据
- **综合表现评分管理**：根据出场时间、进攻、防守等数据综合评定球员表现评分
- **数据规范与查询支持**：支持多维度数据查询与分析

### 2.3 数据源
Kaggle数据集：Premier League Dummy Dataset (2019–2023)
- 90名球员、6支球队（Arsenal, Chelsea, Liverpool, Manchester City, Manchester United, Tottenham）
- 5个赛季（2019-2023），共5482场比赛记录
- 每场比赛记录球员的出场时间、进球、助攻、射门、成功传球、抢断和评分

---

## 三、功能模块划分

根据需求分析，将系统划分为以下三个功能模块：

### 模块1：球员与球队管理
- 球员信息的增删改查（姓名、国籍、位置、出生日期）
- 球队信息的维护（球队名称、城市、主场）
- 球员与球队的归属关系管理

### 模块2：赛季与比赛管理
- 赛季信息维护（年份）
- 比赛信息管理（比赛场次、日期、主客队）
- 赛季与比赛的包含关系

### 模块3：比赛表现管理
- 球员每场比赛的表现数据记录
- 进攻指标：进球、助攻、射门
- 防守指标：抢断
- 综合指标：出场分钟、成功传球、评分

## 五、实体间联系

| 联系名 | 实体1 | 联系类型 | 实体2 | 说明 |
|--------|-------|----------|-------|------|
| 属于 | 球员 | N : 1 | 球队 | 多名球员属于一支球队 |

## 七、E-R图优化分析

### 7.1 数据冗余检查
- **球员信息**：球员姓名、国籍、位置等基本信息只在"球员"实体中存储一次，无冗余
- **球队信息**：球队名称、城市、主场只在"球队"实体中存储一次，无冗余
- **表现数据**：每个球员每场比赛的表现单独存储，无重复
- **结论**：当前设计不存在数据冗余

### 7.2 插入异常检查
- 新增一支球队时，即使该队尚无球员，也可以在"球队"表中正常插入 ✓
- 新增一个赛季时，即使尚未安排比赛，也可以在"赛季"表中正常插入 ✓
- 新增一名球员时，必须先指定所属球队（外键约束），但不能为空 → 可设允许暂时无球队
- **结论**：无插入异常

### 7.3 删除异常检查
- 删除一支球队时，该队所有球员的外键引用会受影响 → 需设置级联策略（SET NULL或限制删除）
- 删除一名球员时，其历史表现数据会丢失 → 如果需要保留历史数据，可采用逻辑删除
- **结论**：需在物理设计中设置合理的级联删除策略

### 7.4 更新异常检查
- 球队改名时，只需修改"球队"表中的一条记录，球员通过外键引用自动更新 ✓
- 球员转会时，只需修改"球员"表中的所属球队外键 ✓
- **结论**：无更新异常

### 7.5 优化建议
1. 可在"球员表现"实体中添加外键字段（球员ID、比赛ID），确保参照完整性
2. 对于频繁查询的评分字段，可考虑建立索引以提升查询性能
3. 历史表现数据建议采用逻辑删除而非物理删除，保留球员完整职业生涯数据

               │

**整合说明：**
- 全局E-R图将三个局部E-R图通过公共实体进行合并
- "球员"实体在模块1和模块3中同时出现，合并为一个实体
- "比赛"实体在模块2和模块3中同时出现，合并为一个实体
- "球队"实体在模块1中与球员关联，在模块2中作为主客队与比赛关联
- 合并过程中未出现命名冲突或结构冲突

## 九、冲突分析

### 9.1 命名冲突
- 各局部E-R图中实体名称一致，无同名异义或异名同义情况 ✓

### 9.2 结构冲突
- "比赛"实体在模块2中有"比赛场次"和"比赛日期"属性，与模块3中的表现数据关联时的属性集一致 ✓
- "球员"实体在各模块中的属性定义完全一致 ✓

### 9.3 约束冲突
- 各实体间的联系类型在全局范围内保持一致，无冲突 ✓

## 十、总结

本实验完成了英超球员2019-2023赛季表现数据管理系统的概念结构设计：
1. 将系统划分为球员与球队管理、赛季与比赛管理、比赛表现管理三个功能模块
2. 识别出5个核心实体：球员、球队、赛季、比赛、球员表现
3. 定义了6个实体间联系，均为1:N类型
4. 分别绘制了3个局部E-R图并整合为全局E-R图
5. 经过优化分析，当前设计不存在数据冗余，无插入/更新/删除异常
